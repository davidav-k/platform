# Сервис обработки задач

## Статус

Реализованы все необходимые функции, связанные с обработкой задач типа MVP: создание, получение информации о задачах, их перечисление и частичное изменение. Сервис запускается, регистрируется в системе Eureka и загружает конфигурацию с сервера Config Server.

## Цель

Управляет задачами в рамках Платформы управления задачами (TMP). Доступ к функциям платформы осуществляется через API-шлюзы по адресам `/api/tasks` и `/api/tasks/**` . Эти запросы направляются к внутренним точкам доступа `/api/v1/tasks` с использованием механизма обнаружения ресурсов Eureka.

`task-service` будет контролировать задачи, их статусы, комментарии, а также историю выполнения задач. Он не имеет права напрямую обращаться к базе данных `user-service` .

Контракт HTTP-API определен в контракте API сервиса Task. Информация о границах сервиса и способах его взаимодействия с другими компонентами системы приведена в разделе, посвященном границам сервиса.

## Стек

*   Java 17
*   Spring Boot 3.4.0
*   Spring Cloud 2024.0.1
*   PostgreSQL 16.x
*   Миграционные пути
*   Maven

## Миграция баз данных

Flyway отвечает за эволюцию схемы обработки задач. Версии изменений хранятся в директории `src/main/resources/db/migration` ; исходная версия MVP находится в директории `V1__task_service_baseline.sql` .

Настройки базы данных во время выполнения программы предоставляются сервером конфигурации `config/task-service-dev.yml` . Имя базы данных для сервисов выполнения задач по умолчанию составляет `tasks_db`.

## Аутентификация

`POST /api/v1/tasks` , `GET /api/v1/tasks` , `GET /api/v1/tasks/{taskId}` , `PATCH /api/v1/tasks/{taskId}` , `PATCH /api/v1/tasks/{taskId}/status` , `PATCH /api/v1/tasks/{taskId}/assignee` и `DELETE /api/v1/tasks/{taskId}` требуют наличия действительного токена доступа в формате JWT. Сервис принимает тот же заголовок `Authorization: Bearer` и файл cookie `access-token` , которые используются гейтвейтом, и самостоятельно проверяет подлинность токена JWT.

Ответственность за выполнение задач контролируется сервером. Значение `createdByUserId` определяется на основе информации, содержащейся в поле “subject” токена JWT. Это значение игнорируется, если клиент включает его в тело запроса.

Что касается операций с задачами: чтения их содержимого, внесения изменений и получения информации об их статусе, то `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` имеют доступ ко всем задачам. Остальные авторизованные пользователи могут работать только с теми задачами, которые они сами создали или которые им поручены. Право на удаление задач имеют только администраторы и создатели задач. Изменение информации, связанной с задачами, также возможно только для администраторов и создателей задач. Само по себе назначение пользователя на выполнение задачи не предоставляет права на её переназначение или удаление. Если идентификатор задачи недоступен, в качестве ответа возвращается значение `404 NOT_FOUND` , аналогичное тому, что возвращается при отсутствии задачи.

Используйте настоящий токен доступа, выданный `user-service` . Не сохраняйте и не фиксируйте никакие токены.

```bash
ACCESS_TOKEN="<access-jwt>"
```

Пример токена-носителя:

```bash
curl -i \
  -H "Authorization: Bearer ${ACCESS_TOKEN}" \
  http://localhost:8080/api/tasks
```

Пример куки:

```bash
curl -i \
  --cookie "access-token=${ACCESS_TOKEN}" \
  http://localhost:8080/api/tasks
```

При обращении к сервису task-service через API-шлюз, непроверенные запросы отклоняются шлюзом ещё до того, как они достигнут самого сервиса task-service. В таких случаях в качестве ответа возвращается значение `401` .

```bash
curl -i http://localhost:8080/api/tasks
```

Пример содержимого сообщения об ошибках:

```json
{
  "error": "Unauthorized",
  "message": "Missing or invalid token"
}
```

## Интеграция уведомлений

Task-service сохраняет изменения задач и в той же транзакции записывает task events в `outbox_events`. Outbox publisher отправляет события в Kafka topic `platform.task-events`, а notification-service читает их и создает `IN_APP` уведомления, если в payload есть получатель.

Реализованные события:

| Use case | Outbox event |
| --- | --- |
| Создание задачи | `TASK_CREATED` |
| Назначение, переназначение или снятие исполнителя | `TASK_ASSIGNED` |
| Изменение статуса | `TASK_STATUS_CHANGED` |

`TASK_CREATED` создает `IN_APP` уведомление `TASK_CREATED` только если в payload есть `assigneeUserId`. Task-service не вызывает notification-service напрямую для task notifications.

Параметры конфигурации:

*   `OUTBOX_PUBLISHER_ENABLED` (по умолчанию `true`)
*   `OUTBOX_PUBLISHER_ADAPTER` (по умолчанию `kafka`)
*   `OUTBOX_PUBLISHER_KAFKA_BOOTSTRAP_SERVERS` (по умолчанию `kafka:9092`)
*   `OUTBOX_PUBLISHER_KAFKA_TOPIC` (по умолчанию `platform.task-events`)

## API

Внешние клиенты должны обращаться к сервису task-service через API-шлюз:

*   Маршрут прохождения: `http://localhost:8080/api/tasks`
*   Внутренний маршрут: `http://localhost:8086/api/v1/tasks`

### `POST /api/v1/tasks`

Создает задачу типа MVP с помощью `CreateTaskUseCase` .

*   Запрос DTO: `CreateTaskRequest`
*   Ответ в формате DTO: `CreateTaskResponse` в стандартной оболочке `data.task` .
*   Аутентификация: обязательна
*   Собственник: `createdByUserId` определяется на основании информации, содержащейся в подписанном JWT-сообщении.
*   Маршрут прохождения: `POST /api/tasks`

Форма тела в ответе:

```json
{
  "code": 201,
  "status": "CREATED",
  "message": "Task created successfully.",
  "data": {
    "task": {
      "taskId": "aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa",
      "title": "Implement login",
      "description": "Create login functionality",
      "status": "NEW",
      "priority": "HIGH",
      "assigneeUserId": "11111111-1111-1111-1111-111111111111",
      "createdByUserId": "22222222-2222-2222-2222-222222222222",
      "createdAt": "2026-06-03T10:15:30Z"
    }
  }
}
```

Хранимый объект `createdByUserId`  является субъектом подписи JWT, прошедшим аутентификацию.

### `GET /api/v1/tasks/{taskId}`

Возвращает одно задание по его уникальному идентификатору UUID через `GetTaskUseCase` .

*   Ответ в формате DTO: `TaskResponse` в стандартной оболочке `data.task` .
*   Аутентификация: обязательна
*   Разрешения на доступ: пользователи с правами администратора могут просматривать любые задачи. Остальные пользователи должны быть либо создателями этих задач, либо лицами, которым они были поручены.
*   Если нет разрешения на доступ возвращается  `404 NOT_FOUND .`
*   Маршрут прохождения: `GET /api/tasks/{taskId}`


### `GET /api/v1/tasks`

Возвращает список задач в формате постраничного списка через `ListTasksUseCase` .

*   Необязательные фильтры: `status` , `priority` , `assigneeUserId` , `createdByUserId`
*   Нумерация страниц: `page` по умолчанию равно `0` , `size` по умолчанию равно `20 .`
*   Сортировка: по умолчанию используется параметр `createdAt,desc` .
*   Форма кривой отклика: стандартная форма с параметрами `data.items` и `data.page .`
*   Аутентификация: обязательна
*   Разрешения на доступ: пользователи с правами администратора могут видеть все задачи; остальные пользователи могут видеть только те задачи, которые они сами создали или которые им поручены.
*   Маршрут прохождения: `GET /api/tasks`

Форма тела в ответе:

```json
{
  "code": 200,
  "status": "OK",
  "message": "Tasks retrieved successfully.",
  "data": {
    "items": [
      {
        "taskId": "aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa",
        "title": "Implement login",
        "description": "Create login functionality",
        "status": "NEW",
        "priority": "HIGH",
        "assigneeUserId": "11111111-1111-1111-1111-111111111111",
        "createdByUserId": "22222222-2222-2222-2222-222222222222",
        "createdAt": "2026-06-03T10:15:30Z",
        "updatedAt": "2026-06-03T10:15:30Z"
      }
    ],
    "page": {
      "number": 0,
      "size": 20,
      "totalElements": 1,
      "totalPages": 1
    }
  }
}
```

### `PATCH /api/v1/tasks/{taskId}`

Частичное обновление задачи с использованием `UpdateTaskUseCase` .

*   Поля, которые можно редактировать: `title` , `description` , `priority` , `assigneeUserId`
*   Поля, которые не были указаны, остаются без изменений.
*   `description` и `assigneeUserId` могут быть удалены с помощью `null .`
*   Изменения статуса обрабатываются отдельно и не принимаются.
*   Пользователи с правами администратора могут изменять любые задачи. Остальные пользователи могут выполнять только действия, связанные с созданием или назначением пользователей для выполнения определенных задач.
*   Чтобы изменить параметры `assigneeUserId` , необходимы полномочия администратора или разрешение на выполнение соответствующих действий.
*   Задачи, которые отсутствуют или недоступны для выполнения, обозначаются как `404 NOT_FOUND .`
*   Маршрут прохождения: `PATCH /api/tasks/{taskId}`


### `PATCH /api/v1/tasks/{taskId}/status`

Изменяет статус задачи с помощью `ChangeTaskStatusUseCase` .

*   Поле для ввода запроса: обязательно должно содержать значение из набора `NEW` , `IN_PROGRESS` , `DONE` или `CANCELLED` .
*   Для MVP допускаются все допустимые изменения типа “из одного перечисления в другое”. Машина состояний при переходе между состояниями не используется.
*   Пользователи с правами администратора могут изменять любые задачи. Другие пользователи могут только создавать новые задачи или назначаться в качестве исполнителей этих задач.
*   Задачи, которые отсутствуют или недоступны для выполнения, обозначаются как `404 NOT_FOUND .`
*   Другие поля, связанные с выполнением задач, остаются без изменений. Обработкой данных, связанных с `updatedAt` , занимается соответствующий сервис.
*   Маршрут прохождения: `PATCH /api/tasks/{taskId}/status`



### `DELETE /api/v1/tasks/{taskId}`

Проводится «мягкое» удаление задачи с помощью `DeleteTaskUseCase` .

*   Пользователи с правами администратора могут удалять любые задачи. Другие пользователи могут удалять только те задачи, которые сами создали.
*   Получатели задач не могут удалять те задачи, которые они сами не создавали.
*   Ответ: `200 OK` со стандартными метаданными ответа и без каких-либо данных.
*   Задачи, которые отсутствуют, недоступны или уже удалены, отображаются как `404 NOT_FOUND .`
*   Удаленные задачи не учитываются при выполнении обычных операций по получению информации о задачах и их перечислении. Кроме того, с удаленными задачами невозможно выполнять никаких действий.
*   Времена удаления данных и идентификаторы участников операций остаются конфиденциальной информацией и не разглашаются в данных, передаваемых в рамках задач.
*   Маршрут прохождения: `DELETE /api/tasks/{taskId}`


### `PATCH /api/v1/tasks/{taskId}/assignee`

Назначает, переназначает или снимает с кого-либо обязанности по выполнению задачи с помощью символа `AssignTaskUseCase` .

*   Поле для ввода запроса: `assigneeUserId` в формате UUID, или явное указание значения `null` для отмены присвоения.
*   Это поле обязательно должно быть заполнено; пустое полео будет отклонено.
*   Пользователи с правами администратора могут назначать любые задачи другим пользователям. Остальные пользователи должны сами создавать задачи.
*   Получатели задач не могут переназначать те задачи, которые они сами не создавали.
*   Задачи, которые отсутствуют, недоступны или были удалены, отображаются как `404 NOT_FOUND .`
*   Существование пользователя не проверяется с помощью `user-service` в данном конечном пункте доступа MVP.
*   Другие поля, связанные с выполнением задач, остаются без изменений. Обработкой данных, связанных с `updatedAt` , занимается соответствующий сервис.
*   Маршрут прохождения: `PATCH /api/tasks/{taskId}/assignee`

## Локальный запуск

### Предварительные требования

*   Сервер конфигурации ( `http://localhost:8888` )
*   Eureka Server ( `http://localhost:8761` )

### Запускать с помощью Maven

```bash
CONFIG_SERVER_URI=http://localhost:8888 \
ACTIVE_PROFILE=dev \
mvn -f backend/task-service/pom.xml spring-boot:run
```

### Запускать с использованием Docker Compose

```bash
docker compose --env-file .env -f compose.yml up -d --build task-service
```

### Проверка состояния

```
GET http://localhost:8086/actuator/health
```

### Создаем и тестируем.

```bash
mvn -B -f backend/task-service/pom.xml test
```

В ходе тестирования хранилища используется PostgreSQL 16.1 через инструмент Testcontainers. При этом проверяются настройки, связанные с использованием инструмента Flyway, информация о маппингах данных, правила формирования последовательностей значений в таблицах, а также ограничения, связанные с структурой базы данных. Для тестирования функциональности и обеспечения безопасности используется система H2 в режиме совместимости с PostgreSQL. Для выполнения всех необходимых команд необходимо, чтобы Docker был запущен. Инструмент Maven Surefire устанавливает версию API Docker как `1.44` , чтобы обеспечить совместимость с Docker Engine 29.
