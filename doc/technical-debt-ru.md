# Технический долг

| Проблема/Вопрос | Воздействие/Последствия | Приоритетность | Планируемый способ устранения неисправности | Статус |
| --- | --- | --- | --- | --- |
| Отсутствует POM-файл агрегатора Maven с правами на выполнение операций с корневым каталогом. | Модули Maven необходимо запускать независимо друг от друга. Кроме того, необходимо строго соблюдать правила, касающиеся охвата процессов тестирования с помощью инструментов CI. | Низкий | Рассматривать агрегатор POM целесообразно только тогда, когда структура сборки становится стабильной. | Открыто |
| Для использования Testcontainers 1.x необходимо настроить совместимость с API Docker, что возможно только при использовании Docker Engine версии 29. | Тесты на интеграцию с бэкендом зависят от компонента api.version=1.44 в инструменте Maven Surefire. | Средний | Обновляйте Testcontainers, когда базовая версия зависимостей Spring Boot становится совместимой с новой версией Testcontainers. После проверки удаляйте все необходимые настройки, связанные с обновлением Testcontainers. | Открыто |
| Проверка секретной информации и защита от неправомерного загруза данных на GitHub не подтверждены. | Хуки репозиториев снижают вероятность возникновения проблем на локальном уровне. Однако GitHub может все равно принимать конфиденциальную информацию, если защита репозитория отключена. | Высокий | Необходимые настройки и параметры управления описаны в документации. Администратор репозитория должен убедиться в правильной настройке механизмов сканирования конфиденциальной информации и защиты данных при их передаче в GitHub. | Частично завершено |
| Правила защиты филиалов не подтверждены. | Возможны слияние репозиториев без предварительной проверки, прямая загрузка изменений в репозиторий, а также слияния, которые не прошли проверку на соответствие стандартам CI. | Средний | Правила, касающиеся использования main и dev , включая необходимые проверки с использованием инструмента Maven, процедуры рассмотрения изменений, способы разрешения возникающих проблем и требования к обновлениям, описаны в соответствующих документах. Администратор репозитория должен настроить эти правила и проверить их на платформе GitHub. | Частично завершено |
| Процедура полного удаления данных пользователя не соответствует утвержденной политике деактивации функций продукта MVP. | Физическое удаление может привести к потере информации об идентичности пользователя. Однако сервисы, связанные с выполнением задач и отправкой уведомлений, продолжают использовать информацию о пользователе. Механизмы аннулирования токенов и управления их жизненным циклом не определены. | Высокий | Согласуйте семантику жизненного цикла взаимодействия пользователя с сервисом с параметром doc/architecture/user-deletion-policy.md . Определите порядок перехода на новую версию API. Позже информацию о состоянии жизненного цикла необходимо будет передавать с помощью событий, связанных с конкретной версией продукта. | Политика определена; процесс реализации начат. |
| Область применения MVP для сервиса обработки задач: нет комментариев к конечным точкам сети. | Управление жизненным циклом задач остается неполным даже после внесения частичных изменений, изменения их статуса, процедур удаления задач и их назначения другим пользователям. | Средний | Необходимо реализовать оставшиеся элементы интерфейса в рамках следующих этапов разработки MVP. | Открыто |
| Авторизация сервиса обработки задач неполная для запланированных операций записи данных. | Реализованные операции чтения, обновления, изменения статуса, удаления и присвоения элементов данных соблюдают правила, касающиеся ролей пользователей и прав на использование этих элементов. Однако для операций с комментариями все еще необходимы дополнительные правила авторизации. | Средний | При выполнении каждой запланированной операции записи необходимо проводить проверку ролей и ответственности участников процесса. | Частично снижено/Уменьшено |
| При назначении задач не проверяется наличие пользователя. | Задачи могут содержать UUID, который не соответствует идентификатору какого-либо активного пользователя. | Средний | Необходимо внедрить механизм проверки подлинности данных, передаваемых между сервисами, либо использовать информацию о жизненном цикле пользователя, хранящуюся в нескольких копиях. | Открыто |
| Синхронный вызов REST-сервиса с передачей уведомлений осуществляется с использованием JWT-токенов, принадлежащих делегированному пользователю. | Создание уведомлений осуществляется на основе принципа «постараемся сделать как лучше». При этом отсутствуют механизмы повторной попытки отправки уведомления или гарантии атомарности операций. Также отсутствует специальная идентификация сервиса, ответственного за обработку уведомлений. | Средний | Заменить на процесс обработки сообщений, основанный на событиях, а также механизмы аутентификации сервисов. | Открыто |
