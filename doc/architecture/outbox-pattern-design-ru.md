# Дизайн шаблона для папки с отправленными сообщениями

## Статус

В данном документе описывается предлагаемая модель коммуникации, основанная на обработке событий, для данной платформы. Речь идет о целевой архитектуре, а не о уже реализованных функциях.

Действующий MVP продолжает использовать существующие синхронные механизмы интеграции в течение всего периода реализации и проверки этапов миграции, описанных в данном документе. Данная схема не предполагает использования брокеров, специальных структур для хранения данных, компонентов для обработки сообщений, а также какой-либо дополнительной инфраструктуры.

## 1\. Формулировка проблемы

MVP использует синхронную связь между сервисами для отправки уведомлений об распределении задач. Сервис `task-service` сохраняет информацию о задаче, после чего вызывает внутренний REST-эндпойнт `notification-service` . Процесс отправки уведомлений не связан с транзакциями, связанными с базой данных задач.

У этой модели есть несколько недостатков:

*   Связь между компонентами системы: задержки в работе сервиса уведомлений или его сбои влияют на процесс выполнения задач. Для преодоления этих проблем необходимы специальные механизмы обработки сбоев.
*   Распространение сбоев: сетевые ошибки, проблемы с истечением времени и другие сбои возникают на пути передачи запроса.
*   Риск двойной записи: сохранение состояния домена и отправка сообщения об интеграции представляют собой отдельные операции. Поэтому одна из них может быть выполнена успешно, в то время как другая — нет.
*   Неопределенность при повторной попытке: звонящий может не знать, была ли обработана просьба, срок действия которой истек. Это может привести к появлению дублирующихся записей.
*   Механизм аутентификации: При отправке уведомления о текущей задаче JWT инициализирующего пользователя передается другой службе.
*   Ограниченная масштабируемость: при добавлении новых потребителей необходимо вносить изменения в код производителя или устанавливать дополнительные зависимости между компонентами системы.

`user-service` также публикует информацию о событиях, связанных с работой локальных приложений на основе технологии Spring, для целей обработки электронных писем. Эти события полезны в рамках одного процесса, но не являются надежными средствами передачи информации между различными сервисами; кроме того, они не сохраняются при сбое процесса.

Платформе необходим способ надежной обработки изменений, касающихся доменов, без необходимости использования распределенных транзакций между базой данных сервиса и посредником для передачи сообщений.

## 2\. Цели

*   Надежно фиксируйте события, связанные с бизнес-операциями, чтобы их можно было использовать в дальнейшей работе.
*   Необходимо сохранить автономию служб и право собственности на базы данных.
*   Снизить степень прямой взаимосвязи между производителями и потребителями во время выполнения программы.
*   Необходимо обеспечить согласованность данных на разных уровнях сервисов.
*   Необходимо разрешить использование таких услуг независимыми потребителями, как службы уведомлений и услуги по проведению аудитов в будущем.
*   Необходимо сделать так, чтобы сбои в процессе публикации, попытки повторной публикации и задержки в доставке данных были видимы для пользователя.
*   Поддерживается обработка дублируемых доставок в режиме, гарантирующем возможность повторной обработки данных. Обработка является идемпотентной.
*   Необходимо предоставлять контракты на обработку событий в версионированном формате, с возможностью их аудита.
*   Позволять постепенную миграцию, не нарушая при этом текущего функционала MVP.
*   Продолжает быть совместим с Java 17, Spring Boot 3.4.x, Spring Cloud 2024.x, PostgreSQL и Flyway.

## 3\. Нецелевые элементы/Элементы, не относящиеся к целям

*   В этой ветке реализуется использование сервисов Kafka, RabbitMQ, а также механизмов обработки сообщений.
*   Реализация распределенных транзакций или процедуры двухфазной подтверждения.
*   Гарантия немедленной согласованности действий различных сервисов.
*   Гарантия однократной обработки данных на всех этапах.
*   Замена баз данных, находящихся в собственности компании, на общедоступную базу данных.
*   Реализация подхода, основанного на использовании событий для формирования итогового состояния системы. Или, другими словами, воссоздание итогового состояния на основе информации, содержащейся в событиях.
*   Необходимо определить значения всех финальных элементов: полей, тем, каналов обмена данными, очередей и т. д.
*   Замена внешних HTTP-интерфейсов и механизмов маршрутизации через API-гейтвейы на системы обмена сообщениями.
*   Использование событий для обработки запросов/ответов, которые требуют немедленного реагирования.

## 4\. Текущая архитектура

### `user-service`

Платформа управляет информацией о пользователях: идентификаторами пользователей, учетными данными, ролями пользователей, процедурами аутентификации, механизмами двухфакторной аутентификации, профилями пользователей и всем циклом жизни учетных записей. При регистрации пользователей генерируется событие, связанное с работой Spring-приложения; это событие используется для обработки электронных адресов пользователей. Других постоянных событий, связанных с процессом интеграции, не генерируется.

### `task-service`

Управляет задачами, их статусом, а также процедурой удаления задач. При создании задачи, которая не была назначена пользователю лично, после её сохранения выполняется синхронный запрос в сервис уведомлений. Для выполнения этого запроса используется токен доступа отправляющего запроса. В случае неудачи отправки уведомления это записывается в логи, но сама задача выполняется как обычно.

### `notification-service`

Управляет уведомлениями, настройками и информацией о статусе доставки сообщений. Предоставляет доступ к авторизованным HTTP-конечным точкам, а также к внутренней конечной точке, которую использует сервис обработки задач для создания системных уведомлений. Не использует события, генерируемые брокером.

### Общедоступная инфраструктура

API-шлюз направляет внешний HTTP-трафик и отклоняет недействительные JWT-токены на этапе их проверки. Сервер конфигураций обеспечивает настройку приложения во время его работы, а сервис Eureka помогает в поиске необходимых сервисов. Каждый сервис, связанный с хранением данных, использует собственную базу данных PostgreSQL и инструменты для управления изменениями в структуре базы данных (например, Flyway). В настоящее время Docker Compose не включает в себя механизм для обмена сообщениями между компонентами приложения.

## 5\. Предлагаемая архитектура, основанная на принципе обработки событий

Каждый сервис, отвечающий за обработку данных, записывает информацию о произошедших изменениях в свою собственную таблицу для хранения данных, в рамках той же локальной базы данных, что и информация об общих изменениях. Отдельный процесс читает эти данные и отправляет их на указанный брокер. Потребители обрабатывают сообщения независимо друг от друга и обновляют только собственные базы данных.

Пример потока уведомлений о задачах:

```text
Client creates assigned task
          |
          v
task-service transaction
  +-----------------------------+
  | insert/update Task           |
  | insert TaskAssigned outbox   |
  +-----------------------------+
          |
          | commit once
          v
task-service outbox publisher
          |
          v
message broker
          |
          v
notification-service consumer
  +-----------------------------+
  | check eventId idempotency    |
  | create Notification          |
  | record processed eventId     |
  +-----------------------------+
          |
          v
notification-service commit
```

API-шлюз, сервер настроек и сервис Eureka продолжают выполнять свои привычные функции. Трафик, связанный с взаимодействием между различными сервисами, является внутренним трафиком и не должен проходить через API-шлюз.

## 6\. Обзор шаблона обработки сообщений, отправляемых пользователями

Продюсер обрабатывает команды, связанные с доменом, следующим образом:

1.  Проверьте команду и данные, необходимые для авторизации, в соответствии с действующими правилами сервиса.
2.  Обновите суммарные данные о деятельности компании в базе данных, находящейся в ведении сервиса.
3.  Создайте запись в папке с отправленными сообщениями, в которой будет отражено событие интеграции в рамках той же локальной транзакции.
4.  Выполняйте все необходимые изменения и записи в формате “outbox” атомарно.
5.  Издатель, работающий в фоновом режиме, просматривает все соответствующие записи, которые ещё не были опубликованы.
6.  Издатель отправляет информацию об каждом событии брокеру.
7.  После подтверждения брокером, издатель отмечает запись как опубликованную или фиксирует ход процесса публикации.
8.  Потребители обрабатывают эти события и вносят соответствующие изменения в свои собственные базы данных.

Одна и та же операция записи в базу данных включает в себя как информацию о совершенных изменениях, так и информацию о намерении опубликовать эти изменения. Сведения, связанные с совершенной операцией, не могут быть потеряны только потому, что брокер не был доступен в момент выполнения операции. Публикатор может попытаться выполнить операцию повторно позже.

Сбой все еще может произойти после того, как информация будет опубликована брокером, но до того, как соответствующая запись будет помечена как опубликованная. В таком случае информация может быть опубликована снова. Поэтому при проектировании системы используется принцип доставки информации хотя бы один раз, а также требуется, чтобы потребители информации были идемпотентными. Система не гарантирует обработки информации ровно один раз.

Две стратегии реализации проекта со стороны издателей можно будет оценить позже:

*   Издательство, занимающееся проведением опросов, использует механизмы блокировки базы данных и обработки данных пакетами ограниченного размера.
*   Передача данных, связанная с записью строк, отправляемых в выходный канал.

При первоначальной реализации следует отдавать предпочтение такому варианту, который позволяет минимизировать затраты на эксплуатацию при одновременном соблюдении требований к пропускной способности и надежности системы. Решение по этому вопросу может быть принято позже.

## 7\. Предлагаемые виды мероприятий

Это лишь предварительные варианты и цели, которые ещё не были реализованы в виде конкретных договоров.

| Событие | Продюсер | Цель |
| --- | --- | --- |
| TaskCreated | task-service | Объявляется о создании специальной группы по вопросам уведомлений, аудита и составления прогнозов на будущее. |
| TaskUpdated | task-service | Объявляются изменения в полях задач, которые можно редактировать. Полезная нагрузка должна указывать измененные поля или предоставлять информацию об изменениях в соответствии с утвержденной структурой данных. |
| TaskStatusChanged | task-service | Объявляется о конкретных изменениях статуса независимо от общих обновлений. |
| TaskDeleted | task-service | Объявляется о возможности «мягкого удаления» данных, что позволит потребителям обновлять информацию без необходимости удалять исходные данные. |
| TaskAssigned | task-service | Уведомляется о назначении, переназначении или отзыве с должности, а также указываются лица, которые занимали данную должность ранее и в настоящее время, при условии одобрения таких действий согласно установленной схеме. |
| UserCreated | user-service | Объявляется о создании уникальной идентификации пользователя. Однако наличие такой идентификации не означает, что аккаунт является активным. |
| UserActivated | user-service | Уведомляем, что согласно правилам жизненного цикла пользовательских аккаунтов, один из аккаунтов стал активным. |
| UserDeactivated | user-service | Объявляется, что аккаунт стал неактивным, при этом идентификатор пользователя остается прежним. |
| NotificationCreated | notification-service | Объявляется о создании уведомлений, касающихся процедур аудита, доставки или предоставления информации потребителям. |

`NotificationCreated` Нельзя использовать его таким образом, чтобы при этом повторно генерировалось то же самое уведомление. Вопросы, связанные с принадлежностью уведомлений конкретному потребителю и их маршрутизацией, должны быть четко определены заранее.

Существующие имена, такие как `TaskCreatedEvent` , в документации, касающейся границ обслуживания, являются временными обозначениями. При последующем определении способа называния элементов схемы взаимодействия необходимо выбрать один из вариантов именования и применять его последовательно, не меняя уже существующие правила использования протокола HTTP.

## 8\. Рекомендации по структуре данных, передаваемых в ходе событий

Стандартная упаковка для отправки товара должна включать в себя:

| Поле | Цель |
| --- | --- |
| eventId | Уникальный по всему миру идентификатор события, используемый для отслеживания действий и обеспечения идемпотентности операций. |
| eventType | Стабильное название логического события, например, TaskAssigned . |
| aggregateId | Стабильный публичный идентификатор измененного агрегата |
| aggregateType | Категории в совокупности, например, TASK или USER . |
| occurredAt | Временная метка в формате UTC ISO-8601, относящаяся к изменению домена |
| version | Версия схемы события; точный формат определяется в соответствии с условиями контракта. |
| payload | Объект, связанный с конкретным событием, и содержащий минимальный набор данных, необходимый потребителям. |

Рекомендуемые дополнительные метаданные для анализа включают имя производителя, идентификатор связи между элементами данных, идентификатор причинно-следственных связей, версию данных и контекст их передачи. Транспортные метаданные, такие как информация о позициях элементов данных в структуре передачи или идентификаторы элементов данных, не должны входить в состав информации, передаваемой в формате, подходящем для переноса данных.

Правила, касающиеся полезной нагрузки:

*   Используйте стабильные уникальные идентификаторы типа UUID. Ни в коем случае не используйте числовые идентификаторы, присваиваемые локально на уровне сервиса.
*   Включать только те данные, которые необходимы известным потребителям.
*   Предпочтительнее использовать факты, связанные с реально произошедшими событиями, чем отдавать команды другим сервисам относительно того, как им изменять свое состояние.
*   Используйте временные метки в формате UTC, а также явные значения из перечисления.
*   По возможности вносите такие изменения, которые не будут противоречить действующим правилам.
*   Версии схем, несовместимые между собой, а также правила перехода между ними.
*   Не следует передавать по сети никакие объекты, модели JPA или внутренние структуры данных, используемые в рамках сервиса.

Потребители должны рассматривать `eventId` как ключ, обеспечивающий свойства идемпотентности. Потребитель должен записывать информацию о успешной обработке в своей собственной базе данных, вместе с информацией о соответствующих изменениях локального состояния. Повторная доставка одного и того же `eventId` не должна приводить к созданию дополнительных уведомлений или к повторению каких-либо неидемпотентных действий.

Обработка заказов не гарантирована во всех случаях. Там, где порядок обработки заказов имеет значение, необходимо собирать события, связанные с одним и тем же заказом вместе. Потребители должны использовать соответствующие правила для определения устаревших или неправильно расположенных событий.

## 9\. Права собственности и обязанности по обслуживанию

### Сервис обработки задач

*   Постоянно является источником информации о состоянии задачи.
*   Создает события задач в транзакции базы данных задач, что приводит к изменению самой задачи.
*   Публикует записи событий, связанных с выполнением задач, через механизм публикации, находящийся в ведении сервиса обработки задач.
*   Задача не ждет завершения процедуры создания уведомления перед своим выполнением.
*   Логика доставки уведомлений не встроена в процесс публикации событий задач.

### Служба уведомлений

*   Использует соответствующие события задач: сначала события, обозначенные как `TaskAssigned` , а затем и другие утвержденные события задач.
*   Применяет настройки уведомлений и создает записи об уведомлениях в рамках собственной транзакции.
*   Удаление дубликатов событий происходит до того, как они могут вызвать какие-либо побочные эффекты.
*   Обеспечивает возможность повторных попыток доставки и отслеживание статуса уведомлений независимо от сервиса обработки задач.
*   Можно публиковать информацию `NotificationCreated` для утвержденных потребителей на последующих этапах обработки данных.

### Обслуживание пользователей

*   Постоянно является источником достоверной информации о состоянии пользователя на протяжении всего цикла его использования продукта.
*   Публикует события жизненного цикла пользователя в рамках той же транзакции, в которой происходит изменение состояния пользователя на соответствующую стадию его жизненного цикла, как только эти действия по изменению состояния пользователя выполняются.
*   Не публикуется никакая информация, касающаяся учетных данных, значений токенов, секретных данных, связанных с механизмами двухфакторной аутентификации, а также лишняя информация о профиле пользователя.
*   Процедура проверки личных данных будет продолжаться до тех пор, пока не будет разработан специальный механизм для их миграции в другую систему.

Каждый сервис самостоятельно управляет своими записями в папке выходящих сообщений, а также процессами их отправки. Ни один централизованный сервис не выполняет действия по записи сообщений в папку выходящих сообщений от имени другого сервиса.

## 10\. Факторы, связанные с надежностью

### Попытки повтора

Издатели и потребители требуют наличия механизма повторных попыток с учетом факторов задержки и неравномерности времени выполнения операций. Политика повторных попыток должна позволять отличать временные сбои от недействительных сообщений или ситуаций, когда запрос отклоняется по причинам, связанным с самой бизнес-логикой.

### Повторная доставка

При условии доставки сообщений хотя бы один раз ожидается их повторное отправление. Клиенты должны обеспечивать идемпотентность своих действий: значения, обработанные в рамках одной транзакции, должны сохраняться, либо должно соблюдаться эквивалентное условие уникальности этих значений.

### Отравляющие сообщения

Сообщения, передача которых постоянно сбивается, не должны мешать нормальной работе системы. В процессе проектирования брокера необходимо предусмотреть специальные механизмы для обработки таких сообщений: отдельные каналы для хранения непереданных сообщений, ограничения на количество попыток их передачи, уведомления операторов, возможность проверки состояния сообщений, а также механизмы их повторной передачи после устранения ошибок.

### Заказы и одновременное выполнение операций

Гарантии, связанные с процедурой заказа, должны действовать в рамках каждого отдельного агрегата, а не в глобальном масштабе. Издатели должны избегать одновременной публикации одной и той же информации в разных агрегатах. Потребители же должны правильно справляться с ситуациями, когда информация доставляется одновременно в несколько агрегатов, а также с ситуациями, когда используются устаревшие версии информации.

### Восстановление и очистка данных издателя

Записи, находящиеся в очереди на отправку, должны иметь четко определенные параметры, касающиеся способа их публикации, информацию о попытках повторной отправки, а также информацию о сроках их хранения. Опубликованные записи следует архивировать или удалять только в соответствии с установленной политикой хранения данных. Записи, которые не могут быть отправлены, необходимо выявлять и восстанавливать.

### Мониторинг

Как минимум, необходимо соблюдать следующее:

*   Количество непроцессированных сообщений и возраст самого старого сообщения, которое ещё не обработано.
*   Объем публикаций, время задержки, количество попыток повторной отправки данных и количество сбоев
*   Доступность брокера, а также время ожидания или длина очереди клиентов
*   Успешность обработки запросов со стороны потребителей, количество попыток повторной обработки запросов, количество уникальных элементов в данных, а также количество недоставленных сообщений.
*   Время задержки от момента отправки данных от `occurredAt` до момента их обработки потребителем

Логи и трассировочные данные должны включать информацию о корреляционном идентификаторе, типе события, источнике данных и идентификаторе агрегированных данных. При этом нельзя записывать конфиденциальную информацию, содержащуюся в этих данных.

## 11\. Решение брокера относительно своей дальнейшей деятельности

В плане развития хранилища упоминается Kafka, но окончательное решение относительно использования этого инструмента или условия его эксплуатации ещё не принято. Для внедрения любого из вариантов необходимо предварительно разработать соответствующие решения для обработки ошибок и подтвердить возможность эффективной работы данного инструмента.

| Учет/Принятие во внимание | Кафка | RabbitMQ |
| --- | --- | --- |
| Основная модель | Надежный лог-файл с учетом позиций потребителей | Кеи-линии и процедуры обмена данными с подтверждением выполнения операций, организованные посредником |
| Повторная запись/Повторный просмотр | Идеально подходит как для воспроизведения прошлых мероприятий, так и для привлечения новых групп потребителей. | Возможно с использованием определенных топологических решений и способов хранения данных. Однако это не подходит для использования в рамках модели первичной очереди обработки запросов. |
| Заказы | Порядок размещения разделов; важен правильный дизайн ключей. | Порядок обработки запросов с учетом особенностей, связанных с одновременной обработкой запросов, попытками их повторной обработки и их повторной доставкой получателям. |
| Маршрутизация | Темы и группы потребителей | Гибкие механизмы обмена данными, ключи маршрутизации и очереди обработки запросов |
| Стандартный размер/фитинг | Потоки событий, несколько независимых потребителей данных, аналитика, длительное хранение данных | Очереди обработки задач, передача данных в формате команд, гибкое распределение задач, упрощенная структура рабочих процессов |
| Вопросы, связанные с эксплуатацией | Количество разделений диска, правила хранения данных, процедуры перераспределения ресурсов, механизмы управления структурой данных | Топология обмена данными/очередей, подтверждения приема данных, ситуации, когда сообщения теряются, рост размера очередей |
| Интеграция сред с помощью технологии Spring | Весна для Apache Kafka | Spring AMQP |

При принятии решения необходимо учитывать такие факторы, как ожидаемая пропускная способность, потребности в повторном выполнении операций, требования к порядку выполнения действий, количество потребителей, уровень профессионализма персонала, ограничения, связанные с хостингом, а также требования к возможности отслеживания происходящих процессов. Точные версии библиотек необходимо выбирать из набора зависимостей Spring Boot 3.4.x. При начале процесса реализации необходимо также проверить эти версии с использованием инструментов Spring Cloud 2024.x.

## 12\. План миграции MVP

### Этап 1: Текущая синхронная интеграция

Сохраните существующий механизм передачи данных от задач к уведомлениям по протоколу REST. Не добавляйте никаких дополнительных компонентов, связанных с брокерами. Используйте существующие тесты в качестве эталона поведения системы. Перед внесением изменений в поведение системы во время ее выполнения необходимо заранее определить правила обработки событий.

### Этап 2: Сохранение сообщений в памяти устройства

Необходимо добавить процедуры миграции данных из папки отправления, находящейся в ведении сервиса, а также создание событий, связанных с выполнением транзакций. В этот период следует продолжать использовать механизм синхронных уведомлений, если только не используется отдельный план обработки данных, разработанный заранее. Необходимо также убедиться, что при откате состояния домена также происходит откат операций по добавлению данных в папку отправления.

### Этап 3: Публикация информации брокером

Необходимо представить выбранного брокера и сервис для публикации данных. Следует осуществлять публикацию событий, сохраняя при этом существующий синхронный механизм передачи данных. Необходимо оценить надежность процесса публикации, степень возникновения дубликатов данных, правильность их порядка передачи, а также время задержки в передаче данных. Следует избегать ситуаций, когда оба механизма публикации приводят к созданию дубликатов уведомлений в производственной среде. Для проверки эффективности решения следует использовать специальные инструменты, такие как механизмы обработки данных без побочных эффектов, специальные настройки или контролируемую среду для тестирования.

### Этап 4: Внедрение решения среди потребителей

Необходимо реализовать механизмы обработки уведомлений, предусматривающие использование надежных записей о выполнении операций, механизмы повторной попытки выполнения задач и обработку сообщений, которые не были доставлены получателям. Также необходимо проверять правильность создания уведомлений на основе событий, а также процессы восстановления после сбоев на всех этапах работы системы.

### Этап 5: Устранение синхронной связи между компонентами

После того, как публикация и использование результатов мероприятия соответствуют установленным критериям приемлемости, необходимо отключить прямые уведомления, отправляемые сервисом task-service, а также процедуру передачи данных в формате JWT. Во время внедрения новых изменений необходимо соблюдать меры по возврату системы в исходное состояние. Также необходимо обновить условия работы сервиса, соответствующие документацию, схемы и провести все необходимые тесты.

События жизненного цикла пользователя могут происходить в разные фазы независимо друг от друга. Сервисам не обязательно включать все типы событий в одной версии продукта.

## 13\. Эскиз проектирования базы данных

Каждый сервис, занимающийся производством контента, в конечном итоге может иметь свою собственную структуру данных, которая может быть представлена в виде таблицы.

```text
outbox_event
- id
- aggregate_type
- aggregate_id
- event_type
- payload
- created_at
- published_at
- status
```

Вероятные проблемы при реализации включают:

*   UUID `id` соответствует `eventId .`
*   Хранение данных в формате JSON с указанием версии схемы
*   Индексы, отражающие статус документа как непубликованного и порядок его создания
*   Количество попыток повторного выполнения операции, время следующей попытки и метаданные о последней ошибке.
*   Оптимистичные или пессимистичные прогнозы/данные метаданных для одновременно работающих публикаторов
*   Хранение и уничтожение опубликованных записей

Для обеспечения свойств идемпотентности на стороне потребителя также необходимы соответствующие записи о обработанных событиях или сообщениях в почтовом ящике. Точная структура этих записей пока не определена. Любые изменения в структуре данных должны осуществляться с использованием новых миграционных скриптов Flyway в соответствующем сервисе. Использование общей базы данных для хранения сообщений или внешних ключей, связывающих разные сервисы, запрещено.

## 14\. Вопросы безопасности

*   В событиях не следует включать пароли, хэш-значения паролей, данные в формате JWT, токены обновления, секретные данные, необходимые для реализации механизмов двухфакторной аутентификации, ключи для верификации, куки и прочие учетные данные.
*   Не следует передавать JWT-токены конечных пользователей с помощью асинхронных сообщений.
*   Предпочтительнее использовать стабильные идентификаторы вместо имён, адресов электронной почты, номеров телефонов и другой личной информации.
*   Персональные данные следует включать в базу данных только в тех случаях, когда это обусловлено требованиями потребителей или правилами хранения таких данных, которые имеют официальное подтверждение.
*   Проводите аутентификацию и предоставляйте разрешения на доступ к ресурсам брокера с использованием учетных данных с минимальными полномочиями.
*   Шифруйте соединения между брокерами, а также хранимые сообщения в соответствии с требованиями к настройкам системы.
*   Ограничьте производителей возможностью проведения мероприятий только на своих собственных площадках, а потребителей – необходимостью покупать подписки именно на те услуги, которые им нужны.
*   Считайте данные, передаваемые в рамках таких событий, ненадежными по своей природе. Необходимо проверять структуру данных, их размер, типы элементов и идентификаторы.
*   Скрывайте конфиденциальные данные из логов, отчетов о деятельности системы, информации, связанной с обработкой нераспечатанных сообщений, а также из панелей управления.
*   Необходимо определить последствия хранения и уничтожения персональных данных до того, как эти данные будут сохранены в системе, особенно в контексте жизненного цикла пользователя и содержания уведомлений.

## 15\. Открытые вопросы

*   Kafka или RabbitMQ? Какие критерии определяют выбор между этими решениями?
*   Следует ли использовать сервисы по сбору опросных данных или изменить способ сбора информации при первой реализации проекта?
*   Что такое правила формирования тем, процедуры обмена данными, механизмы управления очередями обработки запросов, способы разделения данных на части и правила формирования ключей для маршрутизации данных?
*   Как называть мероприятия и как оформлять информацию о версиях схемы?
*   JSON с соблюдением определенных схем, еще один формат сериализации данных, или же использование реестра схем?
*   Необходимы ли правила организации данных по агрегатам и спецификации поведения различных версий агрегатов?
*   Владение недействительными документами, уведомления, проверка информации, воспроизведение данных и их хранение?
*   Сроки хранения сообщений, отправленных пользователями, и информации о завершенных операциях?
*   Как корреляции и контекстные данные будут передаваться через границы между асинхронными процессами?
*   Какой набор инструментов для мониторинга и какие цели на уровне сервисов будут использоваться?
*   Как будет проверяться совместимость схем данных между производителями и потребителями в процессе интеграции?
*   Какое событие является наиболее подходящим кандидатом для первой фазы миграции: `TaskAssigned` или какое-либо другое событие с более низким уровнем риска?
*   Какие критерии соответствия позволяют отказаться от использования синхронной интеграции с REST-сервисами?

## Краткое изложение решения

Предлагаемый подход заключается в использовании сервиса для обработки транзакций, при котором сообщения доставляются брокером по крайней мере один раз, а потребители могут выполнять свои действия несколько раз без изменения результата. Этот подход позволяет сохранить существующую структуру управления данными и интерфейсы HTTP. Кроме того, синхронные интеграции могут быть постепенно заменены на асинхронные после проверки надежности и эффективности их работы. Детали реализации брокера и механизмов обработки транзакций остаются предметом дальнейших дискуссий.
