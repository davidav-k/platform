# Управление конфигурацией

## Право собственности на конфигурацию

Сервер конфигурации является основным источником информации о настройках сервисов во время их выполнения. Для проекта `user-service` настройки, используемые в процессе разработки, поступают с сервера `config/user-service-dev.yml` .

Приоритет настройок для локальной разработки следующий:

1.  Параметры среды, связанные с конфиденциальной информацией, а также настройки, специфичные для данной среды.
2.  Файлы конфигурации сервера, находящиеся в директории `config/` , предназначены для настройки параметров работы сервиса во время его выполнения.
3.  Собраны все необходимые локальные ресурсы для формирования метаданных сервера настройки, а также параметры, предусмотренные по умолчанию и удобные для использования разработчиками.

Настройки, необходимые для работы сервиса во время его выполнения, не должны повторяться в составе архива с кодом сервиса `application.yml` . Информацию о источниках данных, решениях, связанных с технологией JPA, инструменте Flyway, форматах передачи данных JWT, настройках, связанных с использованием куки, функциях по работе с почтой, сервисе Redis, системе управления ресурсами Eureka, настройках логирования, параметрах, связанных с функциональностью сервиса, а также других настройках, специфичных для данного сервиса, необходимо сохранять в соответствующих файлах на сервере конфигурации. Вместо хранения важной информации в коде программы лучше использовать переменные окружающей среды.

Ресурсы, входящие в пакет услуг для пользователей, преднамеренно включают в себя:

*   `application.yml`: `spring.application.name`.
*   `bootstrap.yml` : стандартная настройка профиля `dev` , параметры клиента сервера конфигурации, а также настройка `CONFIG_SERVER_URI` , которую можно изменить по желанию.
*   `application-dev.yml` : маркер, обозначающий профиль локальной настройки, который сохраняется.

## Запуск приложения с использованием Docker Compose

Из корневого каталога репозитория:

```bash
cp .env.example .env
docker compose --env-file .env -f compose.yml up -d --build
./scripts/check-local-stack.sh
```

Docker использует стандартные настройки сети, установленные системой.

*   Сервер конфигурации: `http://config-server:8888`
*   Eureka Server: `http://eureka-server:8761/eureka`
*   Хост PostgreSQL: `postgres`
*   Хост для обработки сообщений по протоколу SMTP в MailHog: `mailhog`

Составьте конфигурации, связанные с элементами `./config` , и загрузите их на сервер конфигураций по адресу `/config` . Сервисы берут необходимые настройки для своей работы непосредственно с сервера конфигураций, как только тот становится доступным для использования.

## Стартап, предоставляющий услуги для пользователей IntelliJ

Запустите PostgreSQL, MailHog, Config Server и Eureka с помощью Docker Compose. Выполните команду `user-service` в IntelliJ, используя профиль `dev` и те же значения, что и для пользователя root `.env` . Однако необходимо учесть следующие настройки хоста:

```text
CONFIG_SERVER_URI=http://localhost:8888
EUREKA_URL=http://localhost:8761/eureka
POSTGRES_HOST=localhost
EMAIL_HOST=localhost
```

Не требуется никаких дополнительных настроек в формате YAML. Сервер конфигурации по-прежнему является источником информации о настройках, необходимых для работы приложения во время его запуска в среде IDE.

## Добавление свойств

При добавлении нового параметра, связанного с выполнением пользовательских сервисов:

1.  Добавьте его в `config/user-service-dev.yml` .
2.  Используйте специальный знак подстановки для переменной окружающей среды в тех случаях, когда значение этой переменной является конфиденциальным или отличается в зависимости от среды выполнения программы.
3.  Добавьте эту переменную в значения `.env.example` и `doc/configuration/env-variables.md` , если разработчики требуют её наличия.
4.  Не добавляйте свойство runtime в архив, в который включается объект `application.yml` .

При добавлении новых сред развертывания необходимо создавать специальные файлы конфигурации для каждой из этих сред. Не следует хранить в коде важные конфиденциальные данные, связанные с производственной средой.
