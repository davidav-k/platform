# Проверка MVP-версии платформы Postman

В этом каталоге хранится полная информация о процессе обработки запросов в рамках среды Postman для локально запущенной версии приложения MVP. Все запросы, поступающие в приложение, обрабатываются через API-шлюз, расположенный по адресу `http://localhost:8080` . Сама коллекция запросов не обращается непосредственно к сервисным портам.

## Файлы

*   `platform-mvp-e2e.postman_collection.json` – Процедура проверки кандидатов на звание MVP
*   `platform-local.postman_environment.json` – Шаблон локальной среды, не требующий наличия учетных данных или токенов.

## Начните процесс сборки.

Из корневого каталога репозитория:

```bash
cp .env.example .env
```

Установите значение `ADMIN_PASSWORD` на локальном уровне и замените все остальные значения `*_change_me` на значения, указанные в `.env` . Затем запустите процедуру проверки стека.

```bash
docker compose --env-file .env -f compose.yml up -d --build
./scripts/check-local-stack.sh
```

API-шлюз находится по адресу `http://localhost:8080` . MailHog находится по адресу `http://localhost:8025` .

## Импорт и настройка

1.  Импортируйте оба JSON-файла в Postman.
2.  Выберите в качестве активной среды опцию “Local – Postman Template”.
3.  Установите значение `adminPassword` в соответствии с значением `ADMIN_PASSWORD` в локальном файле `.env` .
4.  Установите значения для параметров `userEmail` и `userPassword` для обычного пользователя, использующего аккаунт однократно. Пароль должен содержать заглавные и строчные буквы, а также цифры; его длина должна быть не менее восьми символов.
5.  Сохраняйте значение `baseUrl=http://localhost:8080` , если только не произойдут изменения в порту шлюза, указанном в документации.

Не экспортируйте и не сохраняйте настройки среды после выполнения задачи. Сервис Postman заполняет переменные, связанные с токенами, пользователями, задачами и уведомлениями, значениями, характерными для текущей сессии выполнения программы.

## Подготовьте информацию для обычных пользователей.

Администратор системы создается автоматически; для этого используется адрес электронной почты `admin@mail.com` и настройки, указанные в параметре `ADMIN_PASSWORD` .

Новозарегистрированный пользователь с обычными правами доступа не может пользоваться сервисом до момента подтверждения адреса электронной почты:

1.  Запустите процедуру входа в систему с правами администратора.
2.  Зарегистрируйтесь как обычный пользователь (процедура регистрации является идемпотентной).
3.  Откройте приложение MailHog по адресу `http://localhost:8025` и откройте электронное письмо с информацией о регистрации.
4.  Скопируйте значение параметра запроса `key` из ссылки для верификации и вставьте его в поле `verificationKey` .
5.  Запустите процедуру проверки обычного пользовательского аккаунта.
6.  Выполните вход в систему как обычный пользователь, чтобы подтвердить процедуру активации.

Если уже существует активный пользователь, использующий одноразовый пароль, оставьте поле `verificationKey` пустым. В этом случае процедура проверки пропустится, и при регистрации будет сообщено, что такой адрес электронной почты уже существует. Если существующий аккаунт отключен, необходимо получить и ввести его действительный ключ проверки перед выполнением процедуры регистрации.

## Управляйте коллекцией

После подготовки пользователя используйте функцию Collection Runner в Postman и выполняйте скрипты в порядке, указанном в их названиях. Не отключайте функцию сохранения куки. Скрипты также извлекают значения параметров `access-token` и `refresh-token` из заголовков сообщений типа `Set-Cookie` . Кроме того, скрипты обрабатывают данные, хранящиеся в теле сообщений, если такие данные предусмотрены в спецификации ответа от сервера.

Эта коллекция соответствует установленным стандартам:

*   Портал для управления информацией о здоровье и вход для обычных пользователей
*   Задачи для обычных пользователей включают создание, получение, обновление информации о задачах, изменение их статуса, назначение пользователей для выполнения задач и просмотр списка задач.
*   Отказ в предоставлении прав владельца обычному пользователю для выполнения задачи, принадлежащей администратору, как указано в `404 NOT_FOUND .`
*   Видимость действий администратора для всех владельцев задач
*   Автор и администратор удаляют разрешения.
*   Исключение задач, удаленных в бессрочном режиме, из результатов запросов типа “get” и “list”.
*   Список уведомлений о назначенных задачх, предназначенный для администратора.
*   Обновление куки Refresh-Token

## Текущие лимиты по контрактам

*   Отсутствует публичный эндпоинт для выхода из системы. Запросы на выход из системы блокируются и считаются непринятыми.
*   Функции для работы с списками уведомлений и получения информации об эндпоинтах реализованы. Однако механизмы авторизации пользователей при получении уведомлений, а также функция отметки уведомлений как прочитанных отсутствуют. Фильтрация уведомлений осуществляется по параметру получателя; при этом проверяется идентификатор получателя. Система не гарантирует изоляцию данных, что является обязанностью бэкенда.
*   Создание уведомлений о назначении задач выполняется стараниями системы. Если уведомление не было создано, это приводит к сбою в процессе сбора данных. В таких случаях необходимо проверить логи сервисов, отвечающих за обработку задач и уведомлений.
*   Регистрация не может быть полностью автоматизирована с помощью сервиса Gateway. Проверка учетных данных требует передачи специального ключа на сервер MailHog.
