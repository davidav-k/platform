# Проверки состояния здоровья

## Доступные конечные точки

Каждый сервис, созданный с использованием технологии Spring Boot, предоставляет в доступ только концовые точки типа Actuator `health` и `info` . Конфиденциальные концовые точки типа Actuator, такие как `env` , `beans` , `mappings` , `configprops` , `heapdump` и `threaddump` , не являются доступными для использования.

| Обслуживание | URL-адрес страницы, посвященной вопросам здоровья | URL с информацией | Цель |
| --- | --- | --- | --- |
| Сервер настройки | http://localhost:8888/actuator/health | http://localhost:8888/actuator/info | Подтверждается работоспособность приложения, а также наличие и доступность хранилища настроек. |
| Юникорн Сервер | http://localhost:8761/actuator/health | http://localhost:8761/actuator/info | Подтверждается готовность приложения к процедуре обнаружения сервисов. |
| Обслуживание пользователей | http://localhost:8085/actuator/health | http://localhost:8085/actuator/info | Проверяется работоспособность приложения, наличие соединения с PostgreSQL и доступность сервиса SMTP. |
| Сервис обработки задач | http://localhost:8086/actuator/health | http://localhost:8086/actuator/info | Подтверждается работоспособность приложения и наличие соединения с сервером PostgreSQL. |
| Служба уведомлений | http://localhost:8087/actuator/health | http://localhost:8087/actuator/info | Подтверждается работоспособность приложения и наличие соединения с сервером PostgreSQL. |
| API-шлюз | http://localhost:8080/actuator/health | http://localhost:8080/actuator/info | Подтверждается готовность приложения к работе через шлюз. |

Здоровый сервис возвращает HTTP-ответ с общей информацией о результатах обработки запроса:

```json
{"status":"UP"}
```

Информация о состоянии здоровья намеренно ограничена. Встроенные показатели по-прежнему способствуют получению полной картины состояния здоровья.

## Покрытие зависимостей

Эта стратегия использует встроенные показатели состояния работы системы, предусмотренные в фреймворке Spring, там, где уже реализованы необходимые интеграции.

| Зависимость | Подтверждение/Валидация |
| --- | --- |
| PostgreSQL | Показатель работоспособности источника данных для обслуживания пользователей |
| SMTP | Показатель состояния почтовой службы для пользователей |
| Репозиторий конфигураций | Показатель состояния хранилища данных Spring Cloud Config Server |
| Редис | Проверка работоспособности Docker redis-cli ping |
| Эврика! | Конечная точка доступа к информации о состоянии сервера Eureka, а также процедура регистрации вспомогательных сервисов, на которые опирается сервер Eureka. |

В настоящее время сервис для пользователей использует кэш на основе фреймворка Guava. Интеграции с сервисом Spring Data Redis в данный момент отсутствует. Поэтому состояние сервера Redis проверяется независимо с помощью инструмента Docker Compose. Сервер Redis не считается частью компонентов, отвечающих за работу сервиса для пользователей.

## Проверка жизнеспособности стартапа

Docker Compose использует зависимости при запуске приложений с учетом их состояния:

1.  PostgreSQL и Redis используют встроенные механизмы для сбора информации о своей работе.
2.  Сервер конфигурации запускается и должен сообщить значение `UP` .
3.  Сервер Eureka запускается после сервера конфигурации и должен сообщать информацию в формате `UP` .
4.  Сервис обслуживания пользователей запускается после запуска серверов PostgreSQL, Config Server и Eureka Server. Его точка доступа для проверки состояния сообщает о своей готовности к работе только в том случае, если источник данных и настроенный сервер SMTP находятся в рабочем состоянии.
5.  Сервис Task Service запускается после серверов PostgreSQL, Config Server и Eureka Server. Его точка доступа для проверки состояния возвращает значение `UP` , если источник данных находится в рабочем состоянии.
6.  Сервис уведомлений запускается после серверов PostgreSQL, Config Server и Eureka Server. Его точка доступа для проверки состояния возвращает значение `UP` , если источник данных находится в рабочем состоянии.
7.  API-шлюз запускается после сервера настроек, сервера Eureka, сервиса обработки пользователей, сервиса обработки задач и сервиса уведомлений. Он обязан сообщать информацию в формате `UP` .

Сервисы продолжают регистрироваться в системе Eureka, как и раньше. Сервер конфигурации по-прежнему обслуживает нативный репозиторий, монтируемый с адреса `./config` .

## Проверка совместимости с Docker

Начнем процедуру накопления стека:

```bash
docker compose --env-file .env -f compose.yml up -d --build
```

Запустите скрипт локальной проверки с правами только на чтение из корневого каталога репозитория:

```bash
./scripts/check-local-stack.sh
```

Windows PowerShell:

```powershell
.\scripts\check-local-stack.ps1
```

Скрипт сразу же терпит неудачу, если какой-либо необходимый компонент отсутствует. В таком случае выводится строка с кодом ошибки `ERROR:` . Успешная работа скрипта сопровождается соответствующим сообщением.

```text
Local platform stack verification passed.
```

Скрипт проверяет следующее:

1.  Docker CLI, Docker Compose v2, демон Docker, `.env` и `compose.yml` .
2.  Все десять контейнеров типа Compose, которые должны были быть запущены, действительно работают.
3.  Восемь контейнеров, прошедших проверку состояния с использованием Docker, представлены в отчете `healthy` .
4.  PostgreSQL принимает соединения, а Redis отвечает сигналом `PONG` .
5.  Настройка состояния сервера и прав доступа к хранилищам данных.
6.  Состояние сервера Eureka и доступ к реестру.
7.  Состояние сервисов обслуживания пользователей, выполнения задач, отправки уведомлений, а также API-шлюза.
8.  Маршрутизация уведомлений через API-шлюз с использованием ожидаемого непроверенного ответа типа `401 Unauthorized` .
9.  Маршрутизация через API-шлюз к Службе пользователей осуществляется путем отправки запроса на верификацию публичного аккаунта с использованием ключа, который является недопустимым с точки зрения формата UUID.

Устройство для определения маршрута не создает пользователей, не изменяет данные подтверждения, не отправляет электронные письма и не требует ввода учетных данных. В настоящее время устройство `user-service` может возвращать ответ HTTP `200` с сообщением об ошибках при недопустимом значении ключа. При правильной реализации устройство должно возвращать ответ HTTP `400` . Скрипт считает оба вида ответов подтверждением того, что процедура определения маршрута была выполнена корректно.

Для визуального осмотра используйте:

Проверка состояния здоровья:

```bash
docker inspect --format '{{.Name}} {{.State.Health.Status}}' \
  tsp_config tsp_eureka tsp_user_service tsp_task_service \
  tsp_notification_service tsp_gateway tsp_postgres tsp_redis
```

Проверяйте конечные точки непосредственно:

```bash
curl -fsS http://localhost:8888/actuator/health
curl -fsS http://localhost:8761/actuator/health
curl -fsS http://localhost:8085/actuator/health
curl -fsS http://localhost:8086/actuator/health
curl -fsS http://localhost:8087/actuator/health
curl -fsS http://localhost:8080/actuator/health
```

Проверьте, что сервер конфигурации может обслуживать монтируемый репозиторий:

```bash
curl -fsS http://localhost:8888/user-service/dev
```

Проверьте регистрацию в сервисе Eureka:

```bash
curl -fsS http://localhost:8761/eureka/apps
```

## Устранение неисправностей

### Демон Docker недоступен.

Скрипт проверки завершается до того, как происходит проверка контейнеров, если Docker Desktop или служба Docker выключены. Запустите Docker, а затем выполните следующую команду:

```bash
./scripts/check-local-stack.sh
```

### Постгресквел недоступен.

Сервис обслуживания пользователей не может нормально функционировать, если PostgreSQL недоступен, или если установленные параметры, такие как идентификаторы пользователя, адрес хоста, номер порта или имя базы данных, неверны. Проверьте эти параметры.

```bash
docker compose --env-file .env -f compose.yml logs postgres user-service
```

### Redis недоступен.

Redis находится в неисправном состоянии, если `redis-cli ping` не возвращает значение `PONG` . Проверьте:

```bash
docker compose --env-file .env -f compose.yml logs redis
```

### SMTP недоступен.

Сервис обслуживания пользователей не может нормально функционировать, если сервер SMTP, настроенный для его работы, недоступен. При разработке с использованием Docker Compose значение параметра `EMAIL_HOST` должно соответствовать значению `mailhog` , а не `localhost` . Проверьте настройки.

```bash
docker compose --env-file .env -f compose.yml logs mailhog user-service
```

### Сервер конфигурации недоступен.

Проверьте, что контейнер сервера настроек работает нормально, и что образ `./config` успешно монтируется в папку `/config` . Проверка завершена.

```bash
docker compose --env-file .env -f compose.yml logs config-server
curl -fsS http://localhost:8888/user-service/dev
```

### Eureka недоступен.

Сервисы обслуживания пользователей и API-шлюз могут запускаться, но процедура регистрации или обнаружения сервисов может сорваться, если Eureka недоступен. Проверьте:

```bash
docker compose --env-file .env -f compose.yml logs eureka-server user-service gateway
curl -fsS http://localhost:8761/eureka/apps
```

### Сервис обработки задач недоступен.

Сервис обработки задач остается в некорректном состоянии, если PostgreSQL недоступен или сервер настроек не запущен. API-шлюз не будет выполнять проверки состояния сервиса, пока сам сервис обработки задач не станет корректно функционировать. Проверьте:

```bash
docker compose --env-file .env -f compose.yml logs task-service
curl -fsS http://localhost:8086/actuator/health
```

Проверьте, что сервер конфигурации способен обслуживать процедуры настройки сервисов типа “task-service”.

```bash
curl -fsS http://localhost:8888/task-service/dev
```

### Сервис уведомлений недоступен.

Служба уведомлений не может нормально функционировать, если PostgreSQL недоступен или сервер настроек не запущен. API-шлюз ожидает, пока служба уведомлений восстановит свою работоспособность. Проверьте ситуацию.

```bash
docker compose --env-file .env -f compose.yml logs notification-service
curl -fsS http://localhost:8087/actuator/health
curl -fsS http://localhost:8888/notification-service/dev
```
