# Процесс аутентификации

## Обзор процедур аутентификации

MVP использует JWT-записи, подписанные с использованием алгоритма HMAC и не содержащие информации о состоянии системы. Эти JWT-записи выдаются сервисом `user-service` . Браузер получает два файла типа cookie от сервиса `HttpOnly` .

| Куки/Файлы cookie | Цель | Срок службы браузера по умолчанию |
| --- | --- | --- |
| access-token | Проверяет подлинность запросов, защищенных от несанкционированного доступа. | 600 секунд |
| refresh-token | Проводит обновление обоих куки с помощью соответствующего интерфейса. | 7200 секунд |

Оба вида JWT используют символ `jwt.expiration-seconds` , при этом их действие подтверждается символом `JWT_EXPIRATION` . По умолчанию время действия локальных значений составляет `432000` секунд (5 дней). Время действия куки в браузере намеренно меньше, чем время действия подписанного JWT.

API-шлюз принимает значения `access-token` или `Authorization: Bearer <token>` для маршрутов, защищенных шлюзом. Куки и заголовки авторизации передаются следующим слоям системы. Сервис `user-service` самостоятельно проверяет подлинность JWT-токенов, используемых для авторизации, и принимает решения относительно доступа.

## Процедура регистрации

Внешний конечный пункт: `POST /api/users/register`

Внутренний конечный пункт после пересылки данных через шлюз: `POST /api/v1/user/register`

Запрос DTO: `UserRequest`

Последовательность:

1.  Gateway позволяет осуществлять регистрацию пользователей через определенный путь и модифицировать этот процесс при необходимости.
2.  `UserResource.saveUser` проверяет валидность данных, предоставленных `UserRequest` , и вызывает соответствующий сервис.
3.  `UserServiceImpl.createUser` отклоняет существующую электронную почту.
4.  Эта служба использует системного агента `0` для аудита процедур анонимной регистрации. При этом создается пользователь с ролью `USER` , у которого пароль хранится в зашифрованном виде.
5.  Сервис хранит ключ подтверждения и генерирует событие регистрации.
6.  Сервис для обработки электронных писем отправляет ссылку для подтверждения личности пользователя.
7.  `GET /api/users/verify/account?key=...` является публичным данным. `user-service` проверяет подлинность ключа, активирует учетную запись и удаляет записи, связанные с процедурой подтверждения.

## Процесс входа в систему

Внешний конечный пункт: `POST /api/users/login`

Внутренний конечный пункт после пересылки данных через шлюз: `POST /api/v1/user/login`

Запрос DTO: `LoginRequest`

Последовательность:

1.  Gateway позволяет пользователю войти в систему через определенный путь и переадресовывает его дальнейший трафик.
2.  `AuthenticationFilter` считывает и проверяет содержимое JSON-запроса, содержащего учетные данные пользователя.
3.  Фильтр фиксирует попытки входа в систему и передает соответствующую информацию механизму обработки запросов Spring `AuthenticationManager` .
4.  `ApiAuthenticationProvider` Загружает данные пользователя и учетные данные, проверяет состояние учетной записи, а также проверяет зашифрованный пароль.
5.  При успешном выполнении операции фильтр фиксирует факт успешной авторизации.
6.  При отключенной функции двухфакторной аутентификации фильтр генерирует куки и возвращает данные о пользователе в формате DTO. При включенной функции двухфакторной аутентификации фильтр возвращает пользователю подтверждение необходимости ввода данных для двухфакторной аутентификации, при этом куки не генерируются.

Вход с использованием пароля обрабатывается специальным фильтром, а не контроллером.

## Процедура регистрации на курсы MFA

Внешний конечный пункт: `POST /api/users/enable-mfa`

Внутренний конечный пункт после пересылки данных через шлюз: `POST /api/v1/user/enable-mfa`

Последовательность:

1.  Для входа через Gateway и `user-service` необходимо наличие действительного JWT-токена доступа.
2.  `UserResource.enableMfa` считывает данные, принадлежащие авторизованному пользователю `User` .
3.  Контроллер вызывает функцию `UserService.enableMfa` с использованием электронной почты владельца учетной записи.
4.  Эта служба генерирует и хранит секретный ключ для процедуры двухфакторной аутентификации, а также URL-адрес QR-кода, связанного с данным аккаунтом.

Регистрация осуществляется в режиме самообслуживания. Сервис не принимает параметры запросов, связанные с электронной почтой, и не позволяет работать с другими учетными записями. Секретные данные для механизма двухфакторной аутентификации и одноразовые пароли нельзя сохранять в логах.

## Обновление потока данных

Внешний конечный пункт: `POST /api/users/refresh`

Внутренний конечный пункт после пересылки данных через шлюз: `POST /api/v1/user/refresh`

Последовательность:

1.  Сервис Gateway позволяет обновить данные без необходимости использования токена доступа. Это позволяет восстановить сессию доступа, срок действия которой истек.
2.  Для использования функции `UserResource.refreshTokens` необходим куки `refresh-token` .
3.  `JwtServiceImpl` проверяет подпись JWT и информацию о времени регистрации пользователя, после чего загружает данные, относящиеся к самому пользователю.
4.  Контроллер выдает новые куки, необходимые для получения доступа к информации и обновления данных.

Куки, используемые для обновления информации, меняются с каждым разом. Однако на стороне сервера отсутствуют механизмы хранения токенов, списков отозванных куки, а также механизмы контроля за тем, чтобы куки использовались только один раз.

## Процесс выхода из системы

В настоящее время отсутствует публичный интерфейс для выхода из системы. Сервис `JwtService.removeCookie` позволяет явно удалять куки, но этот функционал не реализован через специальные контроллеры. Поскольку токены не сохраняют информацию о состоянии системы, удаление кук не приводит к аннулированию действительных JWT-токенов. Процедура отзыва токенов на стороне сервера по-прежнему считается технической проблемой, которая нуждается в решении.

## Удаление административной учетной записи

Внешний конечный пункт: `DELETE /api/users/{userId}`

Внутренний конечный пункт после пересылки данных через шлюз: `DELETE /api/v1/user/{userId}`

Для удаления пользователя необходимы полномочия пользователя `user:delete` . Сервис выполняет процедуру полного удаления записей, связанных с номером внутреннего идентификатора пользователя. Сервис отклоняет запросы на удаление пользователей с нулевым идентификатором, отсутствующих в базе данных пользователей, а также запросы на самоудаление. Сервис также удаляет информацию о входе пользователя в систему и данные, связанные с его ролями. После этого база данных удаляет все учетные данные и подтверждения, связанные с этим пользователем.

Удаление касается только данных, находящихся в ведении соответствующих служб пользователей. Механизм совместной обработки данных между различными службами отсутствует.

## Процесс проверки JWT-токенов

Защищенная последовательность запросов:

1.  Клиент вызывает маршрут шлюза с использованием обозначения `access-token` или заголовка сообщения.
2.  Gateway `AuthenticationFilter` проверяет подпись JWT, а также информацию о времени создания сообщения.
3.  Gateway добавляет значение `X-Authenticated-User` в целях передачи информации нижестоящим узлам, при этом оригинальный куки-файл или заголовок с данными авторизации сохраняются без изменений.
4.  `user-service` `JwtAuthenticationFilter` Сначала извлекается куки, предназначенные для обеспечения доступа, а затем используется информация, содержащаяся в заголовке запроса.
5.  `JwtServiceImpl` снова проверяет подлинность JWT, загружает информацию о пользователе и определяет полномочия пользователя на основании данных, содержащихся в токене доступа.
6.  Фильтр заполняет поля Spring `SecurityContext` и `RequestContext` .
7.  Контроллеры используют механизмы, связанные с принципом работы Spring; механизмы обеспечения безопасности методов позволяют проверять полномочия пользователей.
8.  `RequestContext` удаляется в `finally` . Обработка ответов на вход в систему, а также процедуры инициализации административных функций также приводят к удалению временных значений, связанных с запросами. Spring Security отвечает за очистку данных, связанных с `SecurityContext` , с помощью своей цепочки фильтров.

Проверка на уровне шлюза представляет собой этап отказа в обработке запросов на ранней стадии. Однако дальнейшая проверка все же необходима, поскольку процедуры авторизации контролируются отдельными сервисами. Кроме того, внутренний трафик не может полагаться исключительно на результаты проверки, проводимой шлюзом `X-Authenticated-User` .

## Компоненты системы безопасности

| Компонент | Ответственность |
| --- | --- |
| Контроллер | Проверяются данные, передаваемые в формате DTO, выполняются вызовы сервисов, возвращаются ответы в формате DTO. |
| Слой обслуживания | Создание учетных записей пользователей, подтверждение данных пользователей, аудит процедур входа в систему, обновление информации о пользователях |
| JwtServiceImpl | Создание, обработка, проверка, извлечение, выдача и удаление JWT-куки |
| Войти в систему AuthenticationFilter | Обработка запросов по протоколу HTTP с использованием собственных паролей для входа |
| JwtAuthenticationFilter | Проверяйте подлинность запросов от защищенных пользователей и управляйте ими. RequestContext |
| Служба поддержки пользователей SecurityConfig | Определяются публичные пути передачи данных, порядок фильтрации информации, механизмы обработки сессий без учета состояния клиента, а также процедуры авторизации. |
| Гейтвей AuthenticationFilter | Отклонять JWT-токены, отсутствующие или недействительные, при попытке доступа к защищенным ресурсам. |
| Гейтвей SecurityConfig | Разрешить маршрутизацию таким образом, чтобы шлюз GlobalFilter мог соблюдать свои правила формирования маршрутов. |

## Стратегия использования куки-файлов

Поведение куки при работе с `auth.cookie` является предсказуемым:

| Недвижимость | Местный дефолт | Примечания |
| --- | --- | --- |
| secure | false | Установите значение AUTH\_COOKIE\_SECURE=true для использования протокола HTTPS. |
| same-site | Lax | Создано намеренно для использования в заданной топологии фронтенда. |
| access-max-age-seconds | 600 | Срок действия куки, связанной с доступом к сайту, в браузере |
| refresh-max-age-seconds | 7200 | Срок действия куки, используемой для обновления данных в браузере |

Оба куки используют параметры `HttpOnly=true` и `Path=/` . Значения куков, данные в формате JWT, токены обновления информации, а также ключи подтверждения личности пользователя ни в коем случае не должны сохраняться в логах.

## Предположения, касающиеся уровня безопасности

*   `JWT_SECRET` находится в формате Base64 и передается через Гейтвей и `user-service` .
*   Внедрение протокола HTTPS позволяет настроить параметр `AUTH_COOKIE_SECURE=true` .
*   Браузер передает учетные данные, необходимые для авторизации при выполнении запросов между разными доменами с использованием куки.
*   Сервисы, работающие на более поздних этапах обработки данных, проверяют значимость JWT-токенов независимо друг от друга. Они не полагаются исключительно на информацию, содержащуюся в заголовках сообщений, как на единственный способ подтверждения авторизации.
*   Защита от CSRF отключена. Включение этой опции снижает риск уязвимостей, но не заменяет необходимость реализации мер защиты от CSRF в каждой конфигурации системы.

## Примечания по переходу на Keycloak в будущем

В ходе будущей миграции на Keycloak необходимо заменить механизмы проверки паролей и выдачи токенов локальным способом на протоколы OAuth 2.0/OpenID Connect. Необходимо также соблюдать правила разделения данных, связанных с авторизацией, от данных, касающихся самих пользователей. Следует тщательно проверять информацию об источнике выдачи токенов и пользователях, а также правильно сопоставлять внешние роли с соответствующими данными в системе. После миграции не стоит полагаться исключительно на информацию, содержащуюся в заголовках запросов, как на единственный источник подтверждения личности пользователя.
