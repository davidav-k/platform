# Контракт API сервиса обработки задач

## Область применения

В данном документе описываются концовые точки API `task-service` . Реализованные концовые точки соответствуют текущей версии продукта MVP. Планируемые концовые точки представляют собой элементы структуры продукта, которые будут реализованы в будущем. Сервис управляет задачами, их статусами, комментариями и историей выполнения задач, как это определено в рамках его функционала.

Все конечные точки:

*   Используйте конверт для ответов, предусмотренный стандартами API.
*   Необходимо наличие действительного токена доступа JWT.
*   Для идентификаторов областей выполнения задач следует использовать строки в формате UUID.
*   Используйте стабильные публичные ресурсы `user-service` и `userId` для обозначения пользователей.

## Реализованы конечные точки взаимодействия.

### `POST /api/v1/tasks`

Создает задачу, принадлежащую авторизованному пользователю. Значение поля `createdByUserId` определяется из поля “subject” в сообщении JWT. Если это значение указано в теле запроса, оно игнорируется.

*   Маршрут прохождения: `POST /api/tasks`
*   Запрос DTO: `CreateTaskRequest`
*   Ответ: `201 CREATED` , в `data.task` содержится `CreateTaskResponse .`
*   Аутентификация: обязательна
*   Побочный эффект уведомлений: при выполнении задачи, которая не была назначена пользователем лично, после её завершения отправляется уведомление. Если уведомление не может быть доставлено, это никак не влияет на выполнение самой задачи.

### `GET /api/v1/tasks/{taskId}`

Возвращает одно задание по его публичному UUID.

*   Маршрут прохождения: `GET /api/tasks/{taskId}`
*   Запрос в формате DTO: отсутствует
*   Ответ: `200 OK` , в `data.task` содержится `TaskResponse .`
*   Аутентификация: обязательна
*   Разрешения на доступ: `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` могут читать информацию о любых задачах. Остальные пользователи могут читать информацию о тех задачах, которые они сами создали или которые им поручены.
*   Задача отсутствует или недоступна: `404 NOT_FOUND`

### `GET /api/v1/tasks`

Возвращает отфильтрованный список задач с возможностью перелистывания.

*   Маршрут прохождения: `GET /api/tasks`
*   Запрос в формате DTO: отсутствует
*   Ответ: `200 OK` содержит `TaskResponse` записей, а `data.page` содержит метаданные, связанные с форматом отображения страниц.
*   Аутентификация: обязательна
*   Разрешения: `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` могут просматривать все задачи. Остальные пользователи могут видеть только те задачи, которые они сами создали или которые им поручены.
*   Необязательные фильтры: `status` , `priority` , `assigneeUserId` , `createdByUserId`
*   Нумерация страниц: `page` (по умолчанию `0` ), `size` (по умолчанию `20` ; максимум `100` )
*   Сортировка: `sort` (по умолчанию `createdAt,desc` )

### `PATCH /api/v1/tasks/{taskId}`

Частичное обновление полей задач, которые можно редактировать.

*   Маршрут прохождения: `PATCH /api/tasks/{taskId}`
*   Запрос DTO: `UpdateTaskRequest`
*   Ответ: `200 OK` , в `data.task` содержится `TaskResponse .`
*   Аутентификация: обязательна
*   Разрешения: `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` могут обновлять любые задачи. Остальные пользователи могут обновлять только те задачи, которые они сами создали или которые им поручены.
*   Задача отсутствует или недоступна: `404 NOT_FOUND`
*   Необходимо указать по крайней мере одно поле, с которым можно работать.
*   Поля, которые не указаны, остаются без изменений. Значения `description` и `assigneeUserId` можно сбросить с помощью `null .`
*   Чтобы изменить параметры объекта `assigneeUserId` , необходимы разрешения от пользователей `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` , либо от создателя этого объекта.
*   Изменения статуса не поддерживаются этим конечным пунктом.

### `PATCH /api/v1/tasks/{taskId}/status`

Изменяет статус задачи независимо от общих изменений, касающихся всех задач.

*   Маршрут прохождения: `PATCH /api/tasks/{taskId}/status`
*   Запрос DTO: `UpdateTaskStatusRequest`
*   Ответ: `200 OK` , в `data.task` содержится `TaskResponse .`
*   Аутентификация: обязательна
*   Разрешения: `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` могут изменять любые задачи. Остальные пользователи могут изменять только те задачи, которые они сами создали или которые им поручены.
*   Задача отсутствует или недоступна: `404 NOT_FOUND`
*   Все допустимые изменения типа “enum-to-enum” разрешены для MVP.
*   Другие поля, связанные с выполнением задач, остаются без изменений. Обработкой данных, связанных с `updatedAt` , занимается соответствующий сервис.
*   История выполнения задач и правила перехода между этапами выполнения задач пока не реализованы.

### `DELETE /api/v1/tasks/{taskId}`

Проводится «мягкое» удаление активной задачи, при этом строка, соответствующая этой задаче, не удаляется из базы данных.

*   Маршрут прохождения: `DELETE /api/tasks/{taskId}`
*   Запрос в формате DTO: отсутствует
*   Ответ: `200 OK` с стандартной метаданными ответа и без поля `data` .
*   Аутентификация: обязательна
*   Разрешения: `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` могут удалять любые задачи. Остальные пользователи могут удалять только те задачи, которые сами создали.
*   Одного лишь назначения пользователя на определенную роль недостаточно для получения права на удаление данных.
*   Задача отсутствует, недоступна или уже удалена: `404 NOT_FOUND`
*   Удаленные задачи не учитываются при выполнении обычных операций по получению информации о задачах и их перечислении. Их нельзя обновить или изменить их статус.
*   `deletedAt` и `deletedByUserId` являются внутренними полями для хранения данных и не предоставляются для использования через интерфейс `TaskResponse .`

### `PATCH /api/v1/tasks/{taskId}/assignee`

Назначает, переназначает или снимает с выполнения определенную задачу.

*   Маршрут прохождения: `PATCH /api/tasks/{taskId}/assignee`
*   Запрос DTO: `AssignTaskRequest`
*   Ответ: `200 OK` , в `data.task` содержится `TaskResponse .`
*   Аутентификация: обязательна
*   Разрешения: `ROLE_ADMIN` и `ROLE_SUPER_ADMIN` могут назначать любые задачи; другие пользователи могут назначать только те задачи, которые сами создали.
*   Одного лишь назначения недостаточно для получения разрешения на передачу полномочий кому-либо другому.
*   Задача отсутствует, недоступна или удалена: `404 NOT_FOUND`
*   `assigneeUserId` должен присутствовать в формате UUID или содержать явное значение `null` , чтобы процедура отмены назначения могла быть выполнена.
*   Этот сервис не проверяет наличие пользователя в данном эндпоинте синхронно.
*   Другие поля, связанные с выполнением задач, остаются без изменений. Обработкой данных, связанных с `updatedAt` , занимается соответствующий сервис.

## Перечисления

### Статус задачи

| Ценность | Описание |
| --- | --- |
| NEW | Новая задача, которая ещё не начата. |
| IN\_PROGRESS | Задача находится в процессе выполнения. |
| DONE | Задача выполнена. |
| CANCELLED | Задача была отменена. |

### ЗадачаПриоритет

| Ценность | Описание |
| --- | --- |
| LOW | Низкий приоритет |
| MEDIUM | Средний уровень приоритета |
| HIGH | Высокий приоритет |

## API будущей истории

`task-service` владеет `TaskHistory` . Однако публичный интерфейс для получения информации из истории операций не предусмотрен первоначальным контрактом. Его следует добавить только после того, как будут определены требования к хранению данных, их доступности и процедурам аудита.
