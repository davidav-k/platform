# Стандарты контрактов API

## Область применения

Эти стандарты применяются к новым API-интерфейсам платформы. Существующие точки входа/выхода данных остаются без изменений. Новые контракты должны соответствовать существующей структуре ответов, при этом необходимо избегать всех известных несоответствий, описанных ниже.

## Версионирование URL-адресов

*   Внутренние сервисные API используют схему обозначения `/api/v1/<resources>` .
*   В новых коллекциях ресурсов для обозначения элементов используется множественное число, например, `/api/v1/tasks` .
*   Внешний трафик от браузеров и пользовательских интерфейсов поступает через API-шлюзы, при этом в адресах не указывается информация о версии, например, `/api/tasks/**` .
*   API, предназначенные исключительно для внутреннего использования, используют префикс `/internal/api/v1/<resources>` . Их нельзя направлять через API-шлюз в общедоступном режиме.

Существующий формат `user-service` является преднамеренным исключением из правил совместимости: его внутренний путь размещения данных соответствует адресу `/api/v1/user/**` .

## Пакет ответных данных

Новые сервисы используют ту же структуру данных, которая уже была определена в рамках проекта `user-service` .

| Поле | Тип | Необходимо. | Описание |
| --- | --- | --- | --- |
| time | Строка | Да. | Время ответа согласно стандарту ISO-8601 |
| code | Целое число | Да. | Код статуса HTTP-запроса |
| path | Строка | Да. | Путь запроса, обрабатываемый сервисом |
| status | Строка | Да. | Название статуса HTTP, например: OK . |
| message | Строка | Да. | Краткое изложение, понятное для человека |
| exception | Строка | Нет. | Краткое описание возможных исключений; не указывается при успешных ответах. |
| data | Объект | Да. | Полезный контент ответа или пустой объект |

Пример успешного применения:

```json
{
  "time": "2026-06-02T12:00:00Z",
  "code": 200,
  "path": "/api/v1/tasks/0b3df21d-f449-4ef1-98f8-2532fe49dd16",
  "status": "OK",
  "message": "Task retrieved successfully.",
  "data": {
    "task": {
      "taskId": "0b3df21d-f449-4ef1-98f8-2532fe49dd16",
      "title": "Prepare release notes"
    }
  }
}
```

## Модель ошибки

Ошибки также передаются в том же формате. Фактический статус ответа по протоколу HTTP должен соответствовать значению `code` . В сообщениях об ошибках не должны содержаться никакие конфиденциальные данные: JWT-токены, пароли, информация о внутренних SQL-запросах или детали выполнения программы.

Пример ошибки:

```json
{
  "time": "2026-06-02T12:00:00Z",
  "code": 404,
  "path": "/api/v1/tasks/0b3df21d-f449-4ef1-98f8-2532fe49dd16",
  "status": "NOT_FOUND",
  "message": "Task not found.",
  "data": {}
}
```

Ошибки проверки приводят к отображению сообщений на уровне отдельных полей в `data` .

```json
{
  "time": "2026-06-02T12:00:00Z",
  "code": 400,
  "path": "/api/v1/tasks",
  "status": "BAD_REQUEST",
  "message": "Provided arguments are not valid",
  "data": {
    "title": "Title must not be blank"
  }
}
```

Стандартное использование статуса:

| Статус | Использование |
| --- | --- |
| 400 BAD\_REQUEST | Недопустимый запрос или неподдерживаемый переход между состояниями. |
| 401 UNAUTHORIZED | Отсутствует или недействительная информация для авторизации |
| 403 FORBIDDEN | У авторизованного пользователя нет необходимых полномочий. |
| 404 NOT\_FOUND | Агрегированные данные отсутствуют или недоступны для пользователя, выполняющего запрос. |
| 409 CONFLICT | Конфликт между государственными структурами или дублирование действий |
| 500 INTERNAL\_SERVER\_ERROR | Неожиданная ошибка сервера |
| 503 SERVICE\_UNAVAILABLE | Необходимый компонент отсутствует. |

## Нумерация страниц

Список принимаемых конечных точек:

| Параметр запроса | Тип | По умолчанию | Правила |
| --- | --- | --- | --- |
| page | Целое число | 0 | Должно быть не менее 0 . |
| size | Целое число | 20 | Должно находиться в диапазоне от 1 до 100 . |
| sort | Строка | Специфично для конкретного конечного точка/пункта связи | Формат: \`, |

Пагинационные форматы данных используются для:

```json
{
  "items": [],
  "page": {
    "number": 0,
    "size": 20,
    "totalElements": 0,
    "totalPages": 0
  }
}
```

## Идентификаторы

*   Новыми идентификаторами являются строки в формате UUID.
*   `taskId` , `commentId` , `assignmentId` и `notificationId` являются уникальными идентификаторами типа UUID.
*   Внутрисервисные ссылки используют исключительно стабильные публичные идентификаторы.
*   Контракты, связанные с выполнением задач и уведомлениями, относятся к пользователю по его публичному идентификатору `userId` , предоставленному пользователем `user-service` . При этом не учитывается основной ключ пользователя в базе данных.
*   Идентификаторы числовых записей во внутренней базе данных не должны пересекать границы между различными сервисами.

## Дата и время

*   В временных метках, используемых в рамках API, применяются форматы данных, соответствующие стандарту ISO-8601. При этом учитывается смещение по времени по отношению к координатной системе UTC. Желательно использовать обозначение вида `Z` .
*   Значения, представляющие собой только дату, форматируются в соответствии со стандартом ISO-8601 `YYYY-MM-DD` .
*   Сервисы хранят данные в формате UTC, а конвертацию проводят только в моменты перехода между различными интервалами времени.

## Аутентификация и авторизация

*   API Gateway проверяет подлинность JWT-токенов на этапе их получения, что является одним из способов предотвращения недопустимых действий.
*   Каждый из сервисов, работающих на последующих этапах обработки данных, самостоятельно проверяет подлинность JWT-токенов.
*   Службы, работающие на последующих этапах обработки данных, не должны считать `X-Authenticated-User` единственным доказательством личности пользователя.
*   Решения относительно предоставления прав доступа принимаются той службой, которая управляет данным ресурсом.
*   Конечные точки, предназначенные исключительно для внутреннего использования, требуют процедуры аутентификации при своей реализации. Они не должны опираться на пользовательские данные, поступающие извне.

## Примечания относительно совместимости на данный момент

*   Обработчики исключений используют установленную структуру передачи данных. Однако некоторые обработчики возвращают само содержимое данных напрямую, без использования структуры `ResponseEntity` . В будущих реализациях необходимо будет убедиться в том, что состояние передачи данных соответствует структуре `code` .
*   В настоящее время API Gateway возвращает более сжатый формат данных в формате JSON при отклонении JWT-запросов, а для некоторых ответов используется простой текстовый формат. Выравнивание данных в формате “энвейлопмент” является частью будущих улучшений API Gateway и не входит в текущие требования к его функционалу.
