# Стратегия миграции базы данных

## Область применения

Каждый сервис по обеспечению непрерывности работы приложений имеет свою собственную базу данных PostgreSQL и инструменты для выполнения процедур миграции данных Flyway.

| Обслуживание | Стандартная база данных | Место миграции | Базовый уровень/Начальные показатели |
| --- | --- | --- | --- |
| user-service | users\_db | backend/user-service/src/main/resources/db/migration | V1\_\_user\_service\_baseline.sql |
| task-service | tasks\_db | backend/task-service/src/main/resources/db/migration | V1\_\_task\_service\_baseline.sql |
| notification-service | notifications\_db | backend/notification-service/src/main/resources/db/migration | V1\_\_notification\_service\_baseline.sql |

Эти миграции являются источником достоверной информации о структуре данных. Hibernate проверяет мигрированные схемы данных и не имеет права создавать или обновлять их. Местная среда Docker Compose, а также тесты на интеграцию используют базу данных PostgreSQL `16.1` .

## Принципы

*   Любые изменения в структуре данных производятся с использованием инструмента Flyway.
*   Маппинги сущностей не создают и не обновляют объекты базы данных.
*   Любые изменения в структуре данных требуют проведения соответствующей процедуры миграции данных.
*   Процедуры миграции выполняются автоматически при запуске каждой службы, отвечающей за сохранение данных.
*   Для новой среды достаточно иметь чистую базу данных PostgreSQL. Не требуется никакой ручной обработки запросов на языке SQL.

## Название/Называние

Используйте имена для миграции, соответствующие версии Flyway:

```text
V2__add_user_last_login.sql
V3__create_refresh_token_table.sql
```

Используйте следующий свободный целочисленный вариант и краткое описание, написанное с прописной буквой только в начале слова.

## Правила

*   Никогда не вносите изменения в процесс миграции после его выполнения в общедоступной среде.
*   Всегда создавайте новые процедуры миграции для внесения дополнительных изменений.
*   Ограничивайте процесс миграции рамками конкретного микросервиса, которому он принадлежит.
*   Перед совершением операций коммитирования необходимо тестировать процесс миграции в локальной среде с использованием PostgreSQL.
*   Сохраните настройки Hibernate с использованием `spring.jpa.hibernate.ddl-auto=validate` .
*   Скрипты инициализации PostgreSQL могут создавать базы данных, предназначенные для использования сервисами, но не имеют права создавать таблицы, принадлежащие этим сервисам. Схемы данных, связанные с сервисами, находятся в ведении инструмента Flyway.

## Существующие базы данных

Каждая из базовых линий `V1` предназначена для создания чистой базы данных сервисов. Что касается уже существующих баз данных, созданных вне рамок Flyway:

1.  Сделайте резервную копию базы данных.
2.  Сравните его структуру с базовой версией структуры, характерной для сервиса, которому он принадлежит.
3.  Необходимо применять все требуемые операции согласования данных с использованием SQL-команд. Эти операции должны выполняться в рамках процедур проверки данных.
4.  Базовый маршрут перелета в версии `1` .
5.  Запустите службу управления данными и убедитесь, что процедура проверки данных с помощью Hibernate прошла успешно.

Не включайте автоматическую настройку `baseline-on-migrate` в настройках приложения. Автоматически определяемая базовая конфигурация может принимать неполные или некорректные схемы данных.

## Стратегия возврата к предыдущему состоянию

Используйте способ корректировки ошибок в процессе миграции. Если процедура миграции выполнена неправильно, восстановите данные с проверенной резервной копии базы данных и создайте новую версию процедуры миграции, которая исправит существующие ошибки. Не изменяйте уже выполненные процедуры миграции. При использовании процедур миграции, имеющих разрушительный эффект на структуру базы данных, необходимо сначала создать резервную копию данных и тщательно проработать план их внедрения.

## Локальная проверка

Для сохранения чистой окружающей среды в нашем регионе:

```bash
docker compose --env-file .env -f compose.yml down
rm -rf infrastructure/docker/pgdata
docker compose --env-file .env -f compose.yml up -d --build
docker compose --env-file .env -f compose.yml logs user-service
docker exec tsp_postgres psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  -c "SELECT version, description, success FROM flyway_schema_history ORDER BY installed_rank;"
```

Тесты на корректность работы хранилищ данных и процессов интеграции также позволяют проверить правильность выполнения процедур миграции в контексте временных баз данных PostgreSQL 16.1 с использованием инструмента Testcontainers. Требования к совместимости с Docker Engine см. в разделе `doc/development-workflow.md` .

Удаление записи `infrastructure/docker/pgdata` приведет к удалению всех данных из локальной базы данных PostgreSQL. Используйте эту команду только в том случае, если необходимо полностью очистить локальную базу данных.
