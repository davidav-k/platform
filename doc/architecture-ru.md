# Архитектура платформы

## Цель

Эта платформа представляет собой решение для управления задачами на этапе MVP. В настоящее время система позволяет осуществлять управление пользователями, задачами, а также обеспечивает функции отправки уведомлений и использования общей инфраструктуры.

Что касается прав собственности на общие ресурсы и правил предоставления услуг в будущем, см. раздел «Границы предоставления услуг».

Что касается решений, связанных с назначением статуса MVP, а также вопросов, связанных с блокировкой учетных записей пользователей, сохранением информации о пользователях между различными сервисами и управлением жизненным циклом пользователей, см. Политику удаления и блокировки учетных записей пользователей.

Что касается предложенной модели обработки транзакций, кандидатов на роль событий, связанных с данными транзакциями, вариантов действий брокера, а также этапов постепенной миграции данных, см. раздел «Проектирование паттерна обработки транзакций».

## Реализовано во время выполнения программы

Стек Docker Compose в настоящее время работает:

| Компонент | Ответственность | Порт |
| --- | --- | --- |
| user-service | Пользователи, роли, процедуры аутентификации, выдача и проверка токенов JWT, механизмы двухфакторной аутентификации, профили пользователей и жизненный цикл учетных записей | 8085 |
| task-service | Цикл жизни задач, ответственные за их выполнение, порядок распределения задач, изменения статуса задач, возможности фильтрации и пагинации данных, а также функция постепенного удаления задач. | 8086 |
| notification-service | Поддержка сохранения уведомлений, их создания, получения, отображения в списке, фильтрации, разделения на страницы, а также интеграция с процедурами создания задач. | 8087 |
| api-gateway | Внешний точка входа, раннее отклонение токенов типа JWT, механизмы маршрутизации, решение проблем, связанных с правилами CORS, а также механизмы аварийного восстановления работы системы. | 8080 |
| frontend | Библиотека Vue 3, предназначенная для использования в производственных целях, развертывается с помощью сервера Nginx. При этом используется механизм обработки запросов в режиме SPA. | 5173 |
| config-server | Нативный репозиторий Spring Cloud Config, монтируемый из ./config | 8888 |
| eureka-server | Регистрация и обнаружение сервисов | 8761 |
| PostgreSQL 16.1 | Поддержка сохранения данных о пользователях, задачах и уведомлениях в отдельных базах данных | 5432 |
| Redis 7 | Работает независимо и проходит регулярную проверку на наличие ошибок; не интегрирован с сервисами, предназначенными для пользователей. | 6379 |
| MailHog | Локальная обработка сообщений по протоколу SMTP и пользовательский интерфейс | 1025, 8025 |
| Zipkin | Контейнер для инфраструктуры локального отслеживания | 9411 |

Реализованные пути запросов следующие:

```text
client -> api-gateway /api/users/**  -> user-service  /api/v1/user/**
client -> api-gateway /api/tasks/**  -> task-service  /api/v1/tasks/**
client -> api-gateway /api/notifications/** -> notification-service /api/v1/notifications/**
```

Фронтенд Vue 3 работает через сервер Nginx в рамках среды Docker Compose или через инструмент Vite во время процесса разработки фронтенда. Для взаимодействия с бэкендом используются только внешние API-шлюзы; прямое подключение к бэкенд-сервисам не происходит. Подробности о структуре фронтенда и инструкции по его запуску приведены в документации к проекту.

Портал проверяет подлинность JWT-токенов на этапе их получения, тем самым предотвращая недопустимый доступ. Службы, работающие на более поздних этапах обработки запросов, также проверяют JWT-токены и принимают решения относительно авторизации пользователя. Подробности процесса авторизации см. ниже.

## Частично настроено

*   Redis и Zipkin работают в локальном режиме. Однако сервис пользователей в настоящее время не интегрирован с Redis. Также процедура интеграции средств отслеживания действий пользователей не описана полностью.

Контракт API для уведомлений описан в документации к API сервиса уведомлений.

## Право собственности на данные

*   `user-service` владеет своей схемой базы данных PostgreSQL (по умолчанию – `users_db` ).
*   `task-service` владеет схемой базы данных PostgreSQL (по умолчанию — `tasks_db` ).
*   `notification-service` владеет своей схемой базы данных PostgreSQL (по умолчанию – `notifications_db` ).
*   Миграции птиц по маршрутам перелета являются важным фактором для всех трех схем обслуживания.
*   Hibernate использует аннотацию `ddl-auto=validate` . При этом Hibernate не создает и не обновляет схему базы данных.
*   Будущие сервисы также должны иметь собственные схемы данных или базы данных.
*   Сервисы не имеют права напрямую обращаться к базе данных другого сервиса.

См. стратегию миграции базы данных.
